# Reference Library

> **⚠️ Disclaimer:** This is a fictional case study created for portfolio and educational purposes only. The organisation, all systems, vendors, and personnel referenced throughout this portfolio are entirely fictional. No real entity is described, referenced, or disclosed.

This folder sits outside the five numbered projects, deliberately. Projects 1 through 5 each work through one system, one scenario, one stage of the AI governance lifecycle. This folder is the reference layer that sits above all five — two companion documents that don't belong to any single system, but instead organise and cross-reference the whole portfolio against external standards.

## Contents

| File | Description |
|---|---|
| `aigp-knowledge-map.md` | Maps every artefact in this portfolio explicitly to the four domains of the IAPP AIGP Body of Knowledge (v2.1), showing exactly where each certification competency is demonstrated in practice |
| `regulatory-standards-crosswalk.md` | A side-by-side crosswalk of the EU AI Act, NIST AI RMF, ISO/IEC 42001, and the OECD AI Principles, showing how the same governance obligation is expressed differently across all four, and where in this portfolio each one is operationalised |
| `ai-law-and-standards-reference.md` | A standalone, comprehensive reference covering the EU AI Act's structure and current implementation timeline, the US/UK/global regulatory landscape, NIST AI RMF and its emerging profiles, the full ISO/IEC AI standards family, and the OECD AI Principles — written to stand on its own as general AI governance knowledge, independent of this portfolio's fictional case study |

## Why This Exists

Five well-built projects demonstrate execution. This reference layer demonstrates something a portfolio of executed projects cannot show on its own: the ability to step back from the work and organise it against the actual professional standards and regulatory frameworks it is meant to satisfy. That stepping-back capability — being able to say precisely which competency a given artefact demonstrates, and precisely how four different frameworks each treat the same underlying obligation — is itself a distinct, advanced governance skill, separate from the skill of producing any one artefact well.
